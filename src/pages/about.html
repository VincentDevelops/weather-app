The Architecture

I've recently been going over some design patterns and thought I'd practice with them. The base of this project lives on the Observer pattern. If you're interested in design patterns I really recommend Head First Design Patterns, it's a great book for beginners. A quick recap if you're not familiar with the Observer patttern:

The Observer Pattern logically a one-to-many dependency between objects so that when one object changes state, all of its dependents are notified and updated automatically. That's basically it.

First of all we have our WeatherData object which is a makeshift interface. It essentially says "I don't care what API you use, just make sure you have these functions". I implemented it using VisualCrossingWeatherData which uses the Visual Crossing API to get information regarding weather at specific locations. VisualCrossingWeatherData is our Subject/Observable and when changed, will update all its subscribers with the new information.

We also have three sections, each one a Display type, CurrentConditionsDisplay, ForecastDisplay, and HighlightsDisplay. Similarly to WeatherData each one of these implements a Display "interface".

Each Display type takes in a WeatherData object and because I've implemented an interface style, we know what functions we'll call to get the data we need. So in this way, it doesn't matter if we're using Visual Crossing, or another API. So long as we implement WeatherData with whichever other API we want to use, it will work. It makes it incredibly simple and although slightly tedious to implement, very easy to modify/reuse.

When a user enters a location and hits enter, or clicks search, our observer updates all our subscribers. Of course it does this after it has ly completed an asynchronous call to the API.

The rest of the project is simply connecting elements to JS in order to change them on updates. The biggest hurdle, personally, was finding weather icons! Thankfully this repo has everything you need! The icons provided are named exactly as the "icon" names Visual Crossing provides on the API call.

There are a few things I'd improve if I revisted the project, specifcially I would have planned the Fahrenheit/Celsius toggle from the beginning instead of adding it near the end. It wasn't difficult to implement, but it would've ended with cleaner code. I'll like refactor some of that. Overall though I'm fairly happy with how it turned out, it was pretty fun actually.
